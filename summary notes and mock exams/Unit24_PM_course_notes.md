## Introduction <a id="introduction"></a>

Engineering projects — whether the construction of a new manufacturing line, the development of a software system or the delivery of a major infrastructure scheme — rarely succeed by chance. Their success depends on systematic, disciplined management: planning what needs to be done, organising the people and resources to do it, monitoring progress and learning from the experience.

Unit 24 of the Cambridge TECHNICALS Level 3 Engineering qualification introduces learners to the discipline of project management as it is applied in professional engineering contexts. The unit is assessed through a single written examination worth 80 marks, structured around a set of realistic case study scenarios. ## Section 1 — LO1: The Stages of Project Management <a id="lo1"></a>

> **Learning Outcome 1:** Understand the stages of project management  
> **Exam Weighting:** 10–30%  
> **Assessment Criteria:** LO1.1, LO1.2, LO1.3, LO1.4

The four stages of project management — initiation, planning, implementation and closure — are **consecutive and sequential**. Each stage must be completed before the next begins. A project manager who skips the planning stage, for example, risks proceeding with inadequate resources, undefined quality targets and unidentified risks (Turner, 2014).

```
  INITIATION --> PLANNING --> IMPLEMENTATION --> CLOSURE
      |               |              |               |
  Define it       Plan it        Deliver it      Close it
```

### 1.1 Project Initiation <a id="lo1-initiation"></a>

> **LO1.1 — Project Initiation**

Project initiation establishes the rationale, boundaries and initial control framework for a project. It answers the fundamental question: *should this project proceed?*

#### Project Proposal

The project proposal sets out the case for undertaking the project. It typically includes:

| Element | Description |
|---|---|
| **Purpose** | Why the project is needed — the problem being solved or opportunity being pursued |
| **Aims** | Broad statements of what the project seeks to achieve |
| **Scope** | The boundaries of the project — what is and is not included |
| **Deliverables** | The tangible outputs or outcomes the project must produce |

**Engineering example:** The HS2 project proposal identified the purpose as increasing rail capacity between London and the Midlands/North, with the scope defining specific route corridors and the deliverable as new high-speed infrastructure (National Audit Office, 2019).

#### Feasibility Study

A feasibility study assesses whether the project is viable before significant resources are committed. It typically evaluates:

- **Technical feasibility:** Can the solution actually be built or implemented?
- **Financial feasibility:** Are the costs justifiable? Will benefits outweigh costs?
- **Time feasibility:** Can the project be completed within required timescales?

Costings are central to this analysis. A return on investment (ROI) calculation is commonly used to justify the project financially.

#### Project Controls

Project controls define the strategies for managing the three core constraints:

```
         QUALITY
           /\
          /  \
         /    \
        /______\
     COST      TIME

   The Project Management Triangle:
   Changing one side affects the others.
```

| Control | Focus |
|---|---|
| **Time** | Meeting milestones and the completion date |
| **Cost** | Staying within budget |
| **Quality** | Meeting agreed specifications and standards |

#### Communication Strategy

The communication strategy specifies how information will be shared throughout the project. This includes reporting formats, reporting frequency, communication channels (e.g. email, meetings, dashboards) and who receives which information (OCR, 2017).

> **Exam tip:** In the examination, questions on project controls often ask *why* controls are important. The answer should link each control type to the consequence of not having it — for example, without time controls, delays occur and costs escalate.

### 1.2 Project Planning <a id="lo1-planning"></a>

> **LO1.2 — Project Planning**

Project planning translates the approved initiation documents into a detailed roadmap for delivery. The plan specifies who does what, with which resources, by when, at what cost and to what quality standard.

#### SMART Objectives

Objectives must be SMART. In the OCR specification, the 'A' stands for **Agreed** (not Achievable):

```
  S - Specific    : Clear and unambiguous
  M - Measurable  : Quantifiable so progress can be tracked
  A - Agreed      : Accepted by all key stakeholders
  R - Realistic   : Achievable given available resources
  T - Time-specific: Deadline or target date defined
```

#### Outline Plan

The outline plan is a summary document that includes: purpose, aims, objectives, scope, constraints, deliverables, targets, communication strategies and controls.

#### Resource Plan

This identifies the physical (equipment, materials), technological (software, machinery) and human resources needed, specifying *what*, *where* and *when* each resource is required.

#### Financial Plan

| Element | Description |
|---|---|
| **Budget spend** | Planned expenditure over time |
| **Costing** | Unit and total cost calculations |
| **Pricing** | Revenue considerations where applicable |
| **Sources of finance** | Internal (retained profit) or external (loans, equity), short/medium/long-term |

#### Stakeholder Analysis

Stakeholders are mapped on a **power-interest matrix**:

```
HIGH  |  Keep Satisfied  |  Manage Closely   |
      |  (low interest,  |  (high interest,  |
POWER |   high power)    |   high power)     |
      |------------------|-------------------|
      |  Monitor         |  Keep Informed    |
      |  (low interest,  |  (high interest,  |
LOW   |   low power)     |   low power)      |
      +------------------+-------------------+
               LOW             HIGH
                       INTEREST
```

**Internal stakeholders** include employees, management and the project team. **External stakeholders** include clients, suppliers, contractors and regulators.

#### Risk Analysis

Risks are identified, assessed for severity and likelihood, allocated to risk owners and mitigated through the contingency plan.

> **Key distinction:** A **risk** is a potential future event that may impact the project. A **hazard** is a source of potential harm. In project management, risk analysis focuses on risks, not hazards.

#### Contingency Plan

Contingency planning allocates additional time and budget to address risks associated with physical resources (e.g. alternative suppliers), technological resources (e.g. backup systems) and human resources (e.g. cover for key personnel).

#### Quality Plan

The quality plan specifies quality targets and the control methods used to verify that deliverables meet those targets (e.g. inspection, testing, audits).

#### Project Schedule

The project schedule lists all activities, their order of completion and their durations. It feeds directly into the project management tools covered in LO3.

### 1.3 Project Implementation <a id="lo1-implementation"></a>

> **LO1.3 — Project Implementation**

During implementation, the project plan is executed. The project manager's role shifts from planning to directing, monitoring and controlling.

Key implementation activities include:

| Activity | Purpose |
|---|---|
| **Execution** | Carrying out planned tasks according to the schedule |
| **Issue logging** | Recording problems as they arise for tracking and resolution |
| **Monitoring and variance analysis** | Comparing actual progress (time, cost, quality) with planned targets |
| **Control and corrective action** | Approving changes and taking action when variance is identified |
| **Frequent reporting** | Issuing regular project execution reports to stakeholders |
| **Quality management** | Managing quality through inspection, testing and audits |

> **Exam tip:** Activities such as "manage project quality", "issue frequent project execution reports" and "use project control techniques" all belong to the **implementation stage**. A common error is placing quality management in the planning stage — quality is *planned* in Stage 2 but *managed* in Stage 3.

### 1.4 Project Closure <a id="lo1-closure"></a>

> **LO1.4 — Project Closure**

Project closure formally ends the project. It ensures deliverables are accepted, documentation is completed and organisational learning is captured.

Key closure activities include:

```
  CLOSURE STAGE
  |
  +-- Obtain formal acceptance of deliverables
  |
  +-- Complete and archive all project documentation
  |
  +-- Produce the final project report
  |
  +-- Conduct a lessons-learned review
       |
       +-- Identify strengths
       +-- Identify weaknesses
       +-- Identify areas for improvement
       +-- Inform future projects
```

**Lessons learned** are a critical output of project closure. They document what went well, what went wrong and what should be done differently next time. They should be used to improve the organisation's future performance and knowledge base (Association for Project Management, 2019).

> **Exam tip:** "Recognise project successes and failures" belongs to the **closure stage**. "Conduct a feasibility study" belongs to **initiation**.

### 2.1 Project Management Roles <a id="lo2-roles"></a>

> **LO2.1 — Project Management Roles**

Each role within the project structure carries defined duties and responsibilities. Learners must understand *who does what* and *why each role exists*.

```
        PROJECT SPONSOR
             |
             v
        PROJECT BOARD
        (Steering Committee)
             |
             v
        PROJECT MANAGER
             |
      +------+------+
      |             |
      v             v
 TEAM LEADER    TEAM LEADER
      |
      v
 TEAM MEMBERS
```

| Role | Key Duties |
|---|---|
| **Project Sponsor** | Authorises the project to begin; secures funding; champions the project at senior level |
| **Project Board / Steering Committee** | Governs overall direction; chairs key meetings; approves major decisions; receives regular reports |
| **Project Manager** | Plans, organises, directs and controls day-to-day project activities; produces the final report; attends steering committee meetings |
| **Project Team Leader** | Supports and reports to the project manager; instructs, organises and motivates their team members |
| **Project Team Members** | Carry out assigned tasks as directed by their team leader |
| **End Users** | Will use the deliverable; provide requirements and feedback |
| **External Stakeholders** | Consultants, contractors, suppliers, regulators — provide expertise or services outside the core team |

> **Exam tip:** When describing the role of a team leader, always reference *both* directions: upwards (reporting to and supporting the project manager) and downwards (instructing and motivating team members). Answers that only state "leads the team" are too vague for credit.

### 2.2 Skills for Effective Project Management <a id="lo2-skills"></a>

> **LO2.2 — Skills for Effective Project Management**

An effective project manager draws on a broad range of interpersonal, technical and organisational skills. The OCR specification identifies the following:

**Planning | Time Management | Communication | Negotiation | Delegation | Leadership | Team Building | Conflict Management | Information Management | Decision-Making | Problem-Solving**

#### Leadership

Leadership is consistently the most heavily examined skill in this unit. Four main leadership styles are defined:

| Style | Description | When to Use |
|---|---|---|
| **Directive** | Manager tells the team what to do and how to do it | New team members; unfamiliar tasks; time-critical situations |
| **Supportive** | Manager focuses on team wellbeing and relationships | Teams experiencing stress or low morale |
| **Collaborative** | Manager involves the team in decision-making | Experienced team with relevant expertise; complex problems |
| **Responsive** | Manager gives autonomy; team self-directs | Highly experienced, self-motivated team; manager steps back |

> **Exam tip:** The 12-mark leadership question requires learners to identify at least three styles, analyse their characteristics and impacts, and evaluate which is most appropriate for the given scenario. Context from the case study is essential for top marks. A response that lists styles without linking them to the specific situation will not achieve Distinction level.

#### Negotiation

Negotiation is the process of reaching mutually acceptable agreements between parties with differing interests. In a project context, a project manager may need to negotiate with:

- The project sponsor (for additional budget or scope changes)
- Resource holders (for access to staff or equipment)
- Suppliers (for costs, timescales or specifications)
- Team members (for task allocation and workloads)

Effective negotiation typically uses a **win-win** approach, seeking outcomes that satisfy both parties rather than one at the expense of the other (Fisher, Ury and Patton, 2011).

#### Team Building

Tuckman's (1965) model of team development describes five stages through which groups progress:

```
  FORMING --> STORMING --> NORMING --> PERFORMING --> ADJOURNING
    |             |           |            |              |
 Getting       Conflict    Rules set    High          Wrap-up
 acquainted    emerges      norms       output
```

Key team-building techniques include establishing rapport, active listening, motivating individuals, celebrating successes and addressing poor performance. **Active listening** involves giving full attention to a speaker, not merely hearing the words — it builds trust and improves communication quality.

**Motivation methods** (examined frequently in 12-mark questions) include:

- Involving team members in decision-making
- Using collaborative leadership styles
- Providing recognition and praise
- Creating opportunities for professional development
- Organising team social events
- Delegating meaningful responsibilities

> **Note:** Financial rewards are generally not accepted as a motivation method in examinations, as they are outside the project manager's direct control.

#### Communication

Communication strategies consider both the message and the channel:

| Channel | Characteristics | Best for |
|---|---|---|
| Formal meeting | Structured, documented | Major decisions, steering committee reports |
| Email | Written record, asynchronous | Instructions, updates, formal notifications |
| Report | Structured written document | Progress updates, final project reports |
| Informal chat | Flexible, immediate | Quick queries, relationship-building |
| Video call | Remote, synchronous | Multi-site teams |

#### Conflict Management

Conflict in project teams is normal. Thomas and Kilmann (1974) identify five conflict-handling modes:

```
  HIGH  |  Accommodating  |  Collaborating  |
        |  (give in)      |  (win-win)      |
ASSERT- |                 |                 |
IVENESS |   Avoiding      | Compromising    |  Competing
        |   (withdraw)    | (middle ground) |  (win-lose)
  LOW   +-----------------------------------+
                LOW           HIGH
                     COOPERATIVENESS
```

In an engineering project context, **collaborating** is generally the most effective long-term approach, as it preserves relationships while resolving substantive issues.

#### Delegation

Delegation involves assigning tasks and associated authority to team members. Effective delegation:

- Matches tasks to individual capabilities
- Provides clear instructions and deadlines
- Grants appropriate authority to act
- Frees the project manager to focus on higher-priority responsibilities

#### Information Management

This skill involves the collection, organisation and interpretation of project data to support cost control, risk mitigation and performance assessment decisions.

### 3.1 Gantt Charts <a id="lo3-gantt"></a>

> **LO3.1 — Gantt Charts**

A Gantt chart is a horizontal bar chart that displays project activities against a timescale. Each bar represents one activity; the length of the bar represents its duration.

#### Key Terms

| Term | Definition |
|---|---|
| **Activity** | A discrete task within the project |
| **Duration** | The time required to complete an activity |
| **Dependent activity** | An activity that cannot start until a preceding activity has finished (finish-to-start dependency) |
| **Concurrent activity** | Two or more activities that can proceed at the same time |
| **Critical activity** | An activity where any delay will delay the entire project |
| **Slack time** | The amount of time by which a non-critical activity can be delayed without affecting the project completion date |
| **Minimum completion time** | The shortest time in which the entire project can be completed |

#### Reading a Gantt Chart

```
Activity |  1  |  2  |  3  |  4  |  5  |  6  |  7  |  8  |
---------+-----+-----+-----+-----+-----+-----+-----+-----+
   A     |[===]|     |     |     |     |     |     |     |
   B     |     |[===]|[===]|     |     |     |     |     |
   C     |     |[===]|[===]|[===]|[===]|     |     |     |
   D     |     |     |     |     |     |[///]|[///]|[===]|

Key: [===] = activity time   [///] = slack time
Time units (e.g. weeks or months)
```

In this example:
- Activity B is dependent on A (starts when A finishes)
- Activity C runs **concurrently** with B
- Activity D has **slack time** in weeks 6–7; it can be delayed up to 2 weeks without affecting project completion
- Activities A, B, C and D (through its dependency chain) are likely on the **critical path**

#### Advantages of Gantt Charts

- Provides a clear visual overview of the project schedule
- Enables identification of concurrent activities, reducing overall project duration
- Highlights critical activities requiring close monitoring
- Identifies slack time, enabling flexible resource allocation
- Facilitates communication of the project schedule to stakeholders

#### Limitations

- Can become very complex on large projects with many activities
- Does not show the logical relationships between activities as clearly as a CPA network
- Does not quantify uncertainty in activity durations

### 3.2 Critical Path Analysis (CPA) <a id="lo3-cpa"></a>

> **LO3.2 — Critical Path Analysis**

Critical Path Analysis is a network-based technique that identifies the sequence of activities — the **critical path** — which determines the minimum project duration. Any delay on the critical path delays the entire project.

#### Network Diagram Notation

Each activity is represented by a node (circle or box) containing:

```
  +-------+-------+
  |  EST  |       |
  +-------+  ACT  |
  |  LFT  |       |
  +-------+-------+

  EST = Earliest Start Time
  LFT = Latest Finish Time
  ACT = Activity label
```

#### Step-by-Step Method

**Step 1 — Forward Pass (left to right):** Calculate the EST for each activity.

```
  EST of a node = maximum (EST + Duration) of all activities leading into it
```

**Step 2 — Backward Pass (right to left):** Calculate the LFT for each activity.

```
  LFT of a node = minimum (LFT - Duration) of all activities leaving it
```

**Step 3 — Calculate Float:**

```
  Float = LFT - EST - Duration
```

**Step 4 — Identify the Critical Path:**

Activities where **Float = 0** are critical. The critical path connects all zero-float activities from start to finish.

#### Worked Example

Consider activities A–E with the following dependencies and durations:

```
  Activity | Depends on | Duration (weeks)
  ---------+------------+-----------------
     A     |     –      |        3
     B     |     A      |        4
     C     |     A      |        6
     D     |     B      |        2
     E     |   C, D     |        3
```

Network:

```
  START --> [A, 3] --> [B, 4] --> [D, 2] --+
                \                           +--> [E, 3] --> END
                 +--> [C, 6] ---------------+
```

Forward pass:
- Node after A: EST = 3
- Node after B: EST = 3 + 4 = 7
- Node after C: EST = 3 + 6 = 9
- Node after D: EST = 7 + 2 = 9
- Node after E: EST = max(9, 9) + 3 = 12  → **Project duration = 12 weeks**

Critical path: **A → C → E** (all have zero float)

B and D have float: LFT at end of D = 9, EST = 7, Duration of D = 2 → Float = 9 - 7 - 2 = 0. Actually in this case B and D also have zero float, so the critical path is A → B → D → E *and* A → C → E (both paths = 12 weeks — there are two critical paths).

> **Exam tip:** When calculating CPA, always show the forward pass and backward pass workings separately. The mark scheme awards method marks for correct working even if an arithmetic error produces a wrong final answer.

### 3.3 PERT — Programme Evaluation and Review Technique <a id="lo3-pert"></a>

> **LO3.3 — PERT**

PERT extends basic scheduling by incorporating uncertainty into activity duration estimates. Rather than a single duration estimate, three estimates are used.

#### The Three Time Estimates

| Symbol | Name | Definition |
|---|---|---|
| **O** | Optimistic time | The shortest realistic duration (best case) |
| **M** | Most likely time | The most probable duration under normal conditions |
| **P** | Pessimistic time | The longest realistic duration (worst case) |

#### The PERT Formula

The **Estimated Expected Time (E)** is calculated as a weighted average:

```
         O + 4M + P
    E = ____________
               6
```

The weighting gives the most likely time four times the influence of the optimistic and pessimistic estimates, reflecting the assumption that the most likely outcome is the best predictor.

#### Worked Example

An engineering activity has the following estimates:

- Optimistic (O) = 4 weeks
- Most likely (M) = 7 weeks
- Pessimistic (P) = 16 weeks

```
    E = (4 + 4×7 + 16) / 6
      = (4 + 28 + 16) / 6
      = 48 / 6
      = 8 weeks
```

The estimated expected duration is **8 weeks**.

#### Why Use PERT?

PERT is particularly valuable for:

- Research and development projects where activity durations are highly uncertain
- Innovation projects with no historical data
- Projects involving new technologies or processes
- Long-duration projects where risks compound over time

#### Comparison of the Three Tools

```
+----------------+-----------+-----------+-----------+
| Feature        |  Gantt    |    CPA    |   PERT    |
+----------------+-----------+-----------+-----------+
| Visual format  |  Bar chart| Network   | Network + |
|                |           | diagram   | formula   |
| Shows sequence |    Yes    |    Yes    |    Yes    |
| Identifies     |  Partly   |    Yes    |    Yes    |
| critical path  |           |           |           |
| Handles        |    No     |    No     |    Yes    |
| uncertainty    |           |           |           |
| Ease of use    |   High    |  Medium   |  Medium   |
| Best for       | Simple/   | Complex   | Uncertain |
|                | comms     | projects  | projects  |
+----------------+-----------+-----------+-----------+
```

### 4.1 Types of Information <a id="lo4-types"></a>

> **LO4.1 — Types of Information**

Information in a project context can be classified in two dimensions:

**Dimension 1 — Nature:**

| Type | Description | Engineering Example |
|---|---|---|
| **Qualitative** | Descriptive, non-numerical; based on opinions, observations or judgements | Stakeholder satisfaction comments; inspector feedback on workmanship |
| **Quantitative** | Numerical; measurable and statistically analysable | Defect rates; cost variance figures; schedule performance index |

**Dimension 2 — Status:**

| Type | Description | Engineering Example |
|---|---|---|
| **Actual** | Data that has already been recorded | Materials cost to date; hours worked this month |
| **Forecasted** | Data projected into the future based on current trends | Predicted total project cost; estimated completion date |

> **Exam tip:** A common question asks for one benefit and one drawback of using forecasted rather than actual data. A benefit is that forecasts enable proactive decision-making. A drawback is that forecasts based on limited data may be inaccurate — particularly early in a project when patterns have not yet been established.

### 4.2 Sources of Information <a id="lo4-sources"></a>

> **LO4.2 — Sources of Information**

| Source | Description | Examples |
|---|---|---|
| **Internal** | Generated within the organisation | Issue logs, RAID logs, budget reports, quality inspection records, previous project files |
| **External** | Obtained from outside the organisation | Industry benchmarks, government statistics, supplier quotations, trade publications, academic research |

Choosing between internal and external sources depends on availability, relevance, cost and the nature of the decision being made. Internal sources are generally more directly relevant to the specific project; external sources may provide useful context or benchmarks.

| **Primary** | Data collected first-hand for a specific purpose | Interviews, surveys, questionnaires, focus groups, observations | Directly relevant; up to date; tailored to need | Time-consuming; potentially costly; small samples may not be representative |

### 4.4 Judging the Validity of Information <a id="lo4-validity"></a>

> **LO4.4 — Judging Validity of Information**

Before using information to make project decisions, a project manager must assess its validity. The OCR specification identifies ten criteria:

```
+----------------+--------------------------------------------------+
| Criterion      | Question to Ask                                  |
+----------------+--------------------------------------------------+
| Integrity of   | Does the information come from a trusted source? |
| source         |                                                  |
| Bias           | Has the information been presented objectively,  |
|                | or is there an agenda behind it?                 |
| Relevance      | Does the information apply to this project's     |
|                | specific context?                                |
| Complexity     | Is the information at the right level of detail? |
| Detail         | Does the information provide sufficient depth?   |
| Currency       | Is the information up to date?                   |
| Quality        | Does the information meet the required standard? |
| Accuracy       | Is the information factually correct?            |
| Reliability    | If the research were repeated, would the same    |
|                | results be obtained?                             |
| Importance     | Is the information significant enough to matter  |
|                | to this particular business?                     |
+----------------+--------------------------------------------------+
```

> **Exam tip:** Matching exercises in the exam ask learners to connect criteria to definitions. The most commonly confused pairs are: **Quality** (information meets required standards) vs **Reliability** (same results if repeated) vs **Integrity of source** (comes from a trusted source). Memorise these distinctions.

### 4.5 Collating and Using Information <a id="lo4-collating"></a>

> **LO4.5 — Collating and Using Information**

Once valid information has been gathered, it must be organised and presented in a form that supports decision-making.

Key steps include:

1. **Organise** — sort, categorise and structure data (e.g. using spreadsheets, databases or RAID logs)
2. **Present** — select an appropriate format:
   - Tables for structured data comparison
   - Bar charts or line graphs for trends over time
   - Frequency polygons or histograms for distribution data
3. **Interpret** — draw conclusions from the data, noting patterns, anomalies and variances
4. **Decide** — use the interpreted information to make an informed recommendation

The **DECIDE model** provides a structured decision-making framework:

```
  D - Define the problem
  E - Establish criteria
  C - Consider alternatives
  I - Identify best option
  D - Develop and implement
  E - Evaluate the outcome
```

> **Exam tip:** Questions on LO4.5 often provide a data table or graph and ask the learner to calculate a value, interpret a trend and make a decision. Always show calculations clearly and state a specific, justified conclusion — do not hedge or give two contradictory answers.

## Section 5 — LO5: How and Why Projects Are Monitored <a id="lo5"></a>

> **Learning Outcome 5:** Understand how and why projects are monitored  
> **Exam Weighting:** 10–30%  
> **Assessment Criteria:** LO5.1, LO5.2

Monitoring is the process of regularly comparing actual project performance against the plan. It is not a one-off activity but a continuous discipline throughout the implementation stage (Kerzner, 2017). Without monitoring, problems accumulate invisibly until they become crises.

[↑ Return to Table of Contents](#table-of-contents)

---

### 5.1 How Projects Are Monitored <a id="lo5-how"></a>

> **LO5.1 — How Projects Are Monitored**

#### Monitoring Methods and Tools

| Method | Description |
|---|---|
| **Actual vs planned comparison** | Comparing the current state of the project (time, cost, quality) against the baseline plan |
| **Variance analysis** | Quantifying the difference between planned and actual performance |
| **Earned Value Analysis (EVA)** | A technique that integrates scope, schedule and cost data |
| **RAG status reporting** | Red/Amber/Green status assigned to key project areas |
| **RAID log** | Risks, Assumptions, Issues and Dependencies tracked in a live register |
| **Issue log** | A record of problems that have occurred and their resolution status |
| **Quality management** | Inspection, testing, audits and control charts used to verify output quality |
| **Control techniques** | Change control, schedule compression, resource reallocation |

#### Earned Value Analysis (EVA)

EVA uses three core metrics:

```
  EV  = Earned Value   (budgeted cost of work performed)
  PV  = Planned Value  (budgeted cost of work scheduled)
  AC  = Actual Cost    (actual cost of work performed)
```

From these, two variances and two performance indices are derived:

```
  Schedule Variance (SV) = EV - PV
    SV > 0 : Ahead of schedule
    SV < 0 : Behind schedule

  Cost Variance (CV) = EV - AC
    CV > 0 : Under budget
    CV < 0 : Over budget

  Schedule Performance Index (SPI) = EV / PV
    SPI > 1 : Ahead of schedule

  Cost Performance Index (CPI) = EV / AC
    CPI > 1 : Under budget
```

**Engineering example:** During the construction of Heathrow Terminal 5, earned value analysis was used to track the cost and schedule performance of over 100 sub-projects simultaneously, enabling early identification of overruns before they became critical (National Audit Office, 2008).

#### Quality Monitoring

Quality monitoring during implementation includes:

- Comparing outputs against the quality plan's targets
- Completing quality control reports (documenting test findings, pass/fail decisions)
- Identifying defects and initiating corrective action
- Statistical process control using control limits (mean ± tolerance)

### 5.2 Why Projects Are Monitored — Internal and External Factors <a id="lo5-why"></a>

> **LO5.2 — Why Projects Are Monitored**

Projects do not operate in a vacuum. Both internal and external factors can cause project outcomes to diverge from the original plan, making continuous monitoring essential.

> **Key rule (OCR-specific):** The classification of internal vs external factors in the OCR mark scheme follows a specific rule. A factor is **internal** if it originates from *within the organisation*. A factor is **external** if it originates from *outside the organisation*. Note: clients and customers are **external** even if the project is an internal project.

#### Internal Factors

Internal factors are within (or partly within) the organisation's control:

| Factor | How It Can Affect the Project |
|---|---|
| **Organisational aims and objectives** | A change in strategic direction may alter project scope or priority |
| **Policies and procedures** | New procurement policies may affect supplier choices or approval timescales |
| **Ethics and CSR commitments** | Sustainability targets may require changes to materials or methods |
| **Resource constraints** | Budget cuts, staff turnover or equipment unavailability disrupt schedules |
| **Poor leadership** | Lack of direction, poor communication, failure to delegate or motivate can cause time creep |
| **Poor planning** | Inadequate schedules, missing contingencies or unclear objectives create implementation problems |
| **Scale and scope changes** | Scope creep adds work beyond the original plan, increasing cost and time |

#### External Factors — PESTLE

External factors are outside the organisation's direct control:

```
  P - Political    : Government policy, political instability, elections
  E - Economic     : Interest rates, exchange rates, inflation, recession,
                     taxation, unemployment levels
  S - Social       : Changing demographics, public attitudes, consumer
                     behaviour
  T - Technological: New technologies, obsolescence, automation, cyber risk
  L - Legal        : New legislation, regulatory changes, compliance
                     requirements, employment law
  E - Environmental: Climate change, environmental regulations, natural
                     disasters, sustainability requirements
```

**Exam example:** In a project to purchase equipment from a Japanese supplier, a change in the £/¥ exchange rate (an **economic** external factor) could cause the actual cost to exceed budget, requiring monitoring and potentially corrective action. This is a direct example of why continuous monitoring matters even when the project plan is sound.

#### The Consequence of Not Monitoring

| Failure to monitor | Likely outcome |
|---|---|
| Not comparing actual vs planned timescales | Time creep — delays accumulate without detection |
| Not tracking costs against budget | Cost overrun — expenditure exceeds budget |
| Not checking quality outputs | Quality erosion — defects reach the customer |

## Section 6 — LO6: Measuring Project Success <a id="lo6"></a>

> **Learning Outcome 6:** Understand how to measure project success  
> **Exam Weighting:** 7–23%  
> **Assessment Criteria:** LO6.1, LO6.2, LO6.3

Measuring success goes beyond simply checking whether the project was delivered on time and within budget. It involves gathering feedback from stakeholders, evaluating outcomes against objectives and using that learning to improve future performance (Association for Project Management, 2019).

[↑ Return to Table of Contents](#table-of-contents)

---

### 6.1 Types of Feedback <a id="lo6-types"></a>

> **LO6.1 — Types of Feedback**

Feedback gathered at project closure can be classified as either **objective** or **subjective**:

| Type | Description | Examples |
|---|---|---|
| **Objective** | Based on measurable facts; verifiable and reproducible | Issue logs; defect counts; cost variance figures; KPI data; complaints analysis (the *process* of analysing complaints classifies and quantifies them) |
| **Subjective** | Based on opinions, attitudes, beliefs, perceptions or reputation | Project board member opinions; client satisfaction ratings; media coverage; industry reputation |

> **Exam tip:** The distinction between objective and subjective is frequently examined. Issue logs are **objective** (they record factual events). Opinions and reputation are **subjective** (they reflect people's perceptions). **Complaints analysis** is a subtle case — while individual complaints are subjective, the *analysis* involves counting and classifying them, making it **objective**.

[↑ Return to Table of Contents](#table-of-contents)

---

### 6.2 Methods to Gather Feedback <a id="lo6-methods"></a>

> **LO6.2 — Methods to Gather Feedback**

Feedback collection methods are classified as **formal** or **informal**:

| Method | Formal / Informal | Key Features |
|---|---|---|
| **Structured meetings** | Formal | Scheduled; agenda set in advance; minutes recorded |
| **Questionnaires / surveys** | Formal | Standardised questions; written record; can be anonymous |
| **Formal reports** | Formal | Documented, structured analysis; permanent record |
| **Interviews** | Formal (if structured) / Informal (if unstructured) | One-to-one; can probe for detail |
| **Informal chat** | Informal | Unscheduled; no set format; no written record; enables observation of body language |
| **Media monitoring** | Informal | Unstructured; external; potentially biased; not directly controlled |

#### Advantages and Limitations

```
+------------------+---------------------------+---------------------------+
| Method           | Advantage                 | Limitation                |
+------------------+---------------------------+---------------------------+
| Questionnaire    | Anonymous; scalable;      | Low response rates;       |
|                  | creates written record    | limited depth of insight  |
| Formal meeting   | Discussion; two-way       | Time-consuming;           |
|                  | communication; recorded   | participants may be guarded|
| Informal chat    | Natural; body language    | No written record;        |
|                  | observable; immediate     | requires physical presence|
| Media coverage   | Wide audience perspective | Biased; not controlled;   |
|                  | captured                  | may be inaccurate         |
+------------------+---------------------------+---------------------------+
```

### 6.3 Uses of Feedback <a id="lo6-uses"></a>

> **LO6.3 — Uses of Feedback**

Feedback collected at project closure serves multiple purposes:

```
  FEEDBACK
      |
      +---> Identify project strengths
      |         --> Replicate in future projects
      |         --> Share best practice across the organisation
      |
      +---> Identify project weaknesses
      |         --> Root cause analysis
      |         --> Process improvement
      |
      +---> Identify lessons learned
      |         --> Update procedures and documentation
      |         --> Inform risk registers for future projects
      |         --> Guide training needs
      |
      +---> Monitor stakeholder satisfaction
      |         --> Improve client relationships
      |         --> Adjust future project approaches
      |
      +---> Drive business improvement
               --> Better forecasting and estimating
               --> Improved organisational knowledge base
               --> Competitive advantage through learning
```

**Why reviewing lessons learned is important:** Lessons learned ensure that similar mistakes are not repeated in future projects. They contribute to organisational knowledge, improve the accuracy of future plans and can be used to identify training needs, recommend procedural changes and enhance the organisation's project management capability (Association for Project Management, 2019).

> **Exam tip:** A common 1-mark recall question asks for one reason why it is important to review lessons learned. Any of the following are acceptable: to avoid repeating mistakes; to improve future projects; to inform future risk analysis; to update procedures.

> A common 12-mark extended question asks learners to evaluate how feedback can drive business improvement. This requires: identifying specific uses of feedback (L1); explaining how each use works (L2); analysing advantages and disadvantages (L3); and making a justified overall judgement about which use is most valuable in context (L4).

### Response-Level Questions (8–12 marks)

Extended response questions use **levels of response** marking. To achieve the highest level:

```
  Level 1 (1-3 marks) : Identify / list without explanation
  Level 2 (4-6 marks) : Explain how or why
  Level 3 (7-9 marks) : Analyse advantages, disadvantages, impacts
  Level 4 (10-12 marks): Evaluate with justified, contextual judgement
```

**To access Level 4:**
1. Discuss at least three relevant points
2. Analyse the advantages and disadvantages of each
3. Make a clear, justified overall judgement
4. Link responses directly to the case study context

### Common Examination Errors to Avoid

| Error | Correction |
|---|---|
| Placing quality management in the planning stage | Quality management is **implementation**; quality *planning* is planning |
| Writing "Achievable" for the SMART 'A' | OCR uses **Agreed** |
| Classifying clients as internal stakeholders | Clients are always **external** |
| Confusing reliability and quality (validity criteria) | Quality = meets required standard; Reliability = same result if repeated |
| Answering about financial rewards for motivation | Generally **not accepted** as within project manager's control |
| Vague answers ("it would cause problems") | Always state the specific consequence with a causal link |
| Ignoring the case study context | Always reference case study details for top-mark answers |

```
  EVA Variances:        SV = EV - PV
                        CV = EV - AC

  EVA Indices:          SPI = EV / PV
                        CPI = EV / AC
```

## Conclusion <a id="conclusion"></a>

Project management is not simply a set of techniques to be memorised — it is a discipline that connects planning, people, tools and continuous learning into a coherent approach to delivering complex engineering work successfully. The six learning outcomes in Unit 24 reflect this holistic view:

- **LO1** provides the structural framework: initiation, planning, implementation and closure are the backbone of every project.
- **LO2** recognises that projects are delivered by people: roles define accountability and skills determine effectiveness.
- **LO3** equips engineers with practical scheduling tools: Gantt charts for visualisation, CPA for critical path identification and PERT for handling uncertainty.
- **LO4** develops information literacy: knowing what data to gather, where to find it, how to validate it and how to use it to make sound decisions.
- **LO5** embeds the monitoring mindset: understanding both *how* to track a project and *why* tracking is necessary in the face of internal and external pressures.
- **LO6** closes the learning loop: measuring success, gathering feedback and applying lessons learned ensures that each completed project makes the organisation better at the next one.

Learners who demonstrate a command of all six outcomes — and who can apply them in context — will be well placed to achieve strong examination results and to enter professional engineering practice with a solid foundation in project management.
